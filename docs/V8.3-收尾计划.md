## 🔴 阶段一：数据补全（预计 1.5h）

### 1.1 数据库补 2022 年历史数据

**问题**：当前数据库仅覆盖 2023-2026 年。评测中 2022 年的所有 `required_numbers` 都被移除（数据库无数据），导致多年份趋势分析不完整。

**影响**：成长性分析（B02/B03、E02 等题）无法验证 2022→2023 的变化。

**方案**：运行 `financial_populator.py --year 2022` 从 AKShare 拉取 20 家公司的 2022 年财务数据。

**文件**：`backend/db/financial_populator.py`

**验收**：数据库中存在 20 家公司 × 2022 年的 58 个指标数据。

### 1.2 RAG 文档库扩充

**问题**：当前 ChromaDB 仅 3,042 chunks / 10 个文档（8 份年报 + 1 份研报 + 1 份测试）。缺少：
- 招股说明书（业务模式深度解读）
- 季报/半年报（高频数据点）
- ESG/可持续发展报告
- 更多券商深度研报
- 行业白皮书/政策文件

**方案**：
1. 上传已有的几家核心公司（茅台/比亚迪/宁德）的招股书和更多研报
2. 在文档上传页面提示用户"建议上传年报 PDF、研报 .docx 以获得更全面的分析"
3. 暂不追求数量，优先保证质量——已有的 10 份文档覆盖了核心分析场景

**文件**：`data/documents/`（用户上传）

**验收**：ChromaDB chunks ≥ 5,000，文档类型 ≥ 3 种（年报+研报+招股书）

## 🟡 阶段二：评测清理（预计 1h）

### 2.1 清理评测数据中无意义的测试用例

**问题**：`rag_questions.json` 中 R14 "比亚迪2024年的员工满意度调查结果如何"（年报不可能有）和 R15 "？？？。。。"（乱码）作为边界测试无实际价值。

**方案**：
- 将 R14/R15 移到单独的 `rag_questions_boundary.json` 文件（边界测试专用）
- 主评测集 `rag_questions.json` 保留 48 道有效题
- 在 quick_eval.py 中增加 `--include-boundary` 参数可选择性运行边界测试

**文件**：`evaluation/data/rag_questions.json`

**验收**：主评测集无无效测试数据。

### 2.2 为 RAG 评测集补充关键词标注（可按批做）

**问题**：50 道题全部缺少 `expected_keywords`，导致关键词评测不可用。

**方案**：分批评注，本阶段先标注 15 道核心题（原因解读 + 风险识别 + 业务理解类）：

```
每道题补充格式:
"expected_keywords": ["关键词1", "关键词2", ...]

示例（R01 "茅台毛利率下降原因"）:
"expected_keywords": ["毛利率", "下降", "成本", "营业成本", "同比"]
```

**文件**：`evaluation/data/rag_questions.json`

**验收**：≥15 道题有关键词标注，关键词评测可用。

### 2.3 修复 full_eval.py 的 GBK 编码问题

**问题**：`full_eval.py` 用 `subprocess.run(capture_output=True)` 无 `encoding` 参数，Windows 下默认 GBK 解码 UTF-8 字符导致 `UnicodeDecodeError` 崩溃。

**方案**：在 `subprocess.run()` 中添加 `encoding="utf-8", errors="replace"`。

**文件**：`evaluation/full_eval.py`

**验收**：`python evaluation/full_eval.py --light` 能完整跑通三模块评测。

## 🟢 阶段三：优化打磨（预计 1h）

### 3.1 Agent 现金流准确率修复（88.9% → 95%+）

**问题**：现金流类 3 道题中 `经营现金流` 和 `自由现金流` 的数值匹配不稳定——数据值在元/亿元单位转换有偏差。

**方案**：在 `score_number_accuracy_v2` 中增加对现金流特殊指标的单位适配（`operating_cf` 等指标名识别）。

**文件**：`evaluation/agent/bench_agent.py`

**验收**：现金流类型 ≥ 95%。

### 3.2 RAG 检索延迟优化（26.0s → 5s）

**问题**：全量模式下 CrossEncoder CPU 重排平均耗时 26 秒，用户体验差。虽然可通过 `EVAL_LIGHT=1` 跳过重排，但默认应该更快。

**方案**：
- 检查 CrossEncoder 模型是否正确复用（怀疑每次查询重新加载导致耗时翻倍）
- 在 `hybrid_search.py` 中增加重排超时保护（>5s 则跳过重排用 RRF 结果）
- 默认 `EVAL_LIGHT=1` 或增加智能判断（query < 20 字走轻量）

**文件**：`backend/rag/hybrid_search.py`、`backend/config.py`

**验收**：全量模式平均延迟 ≤ 8s，轻量模式 ≤ 3s。

### 3.3 ChromaDB 熔断器（低优先级）

**问题**：V8.2 计划要求但未实现。ChromaDB 是本地的，熔断意义有限。

**方案**：在 `hybrid_search.py` 的 CrossEncoder 加载处增加熔断器（作为网络模型加载的保护）。

**文件**：`backend/rag/hybrid_search.py`

**验收**：CrossEncoder 连续加载失败 3 次后熔断。
