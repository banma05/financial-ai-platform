## 🔴 阶段一：核心质量防线（预计 2h）

### 1.1 Agent 准确率修复（🔴 最关键）

**问题**：数字准确率仅 39.7%，12/15 生成报告。"成长性"和"风险扫描"模板表现最差，跨公司查询薄弱。

**根因分析**：
- Planner 拆解任务时键名匹配残差（三家公司指标名不一致）
- Reporter 提取数字时 LLM 幻觉（非结构化文本中找不到精确值）
- 跨公司查询时 executor 未传递正确的公司上下文

**修复方案**：
- 优化 Planner prompt：在模板中注入可用指标键名列表
- Reporter 增加数值校验：从 SQL 结果中精确提取，禁止 LLM 自行估算
- 超时策略：Agent 单任务超时从 30s 降到 15s，总体 45s

**验收标准**：数字准确率 ≥ 80%，报告生成率 100%

**文件**：`backend/agent/planner.py`、`backend/agent/reporter.py`、`backend/agent/executor.py`

### 1.2 RAG 幻觉率降低

**问题**：33.3% 的回答包含来源中不存在的短语。

**根因**：LLM 生成答案时未严格约束"只能使用检索到的原文片段"。

**修复方案**：
- Prompt 强化：明确标注"如果检索结果中没有相关数据，请直接说'知识库中暂无相关信息'，不要编造"
- 增加后处理校验：用 BM25 检查生成文本中的数字是否出现在来源文档中
- source 引用格式改为 inline `[1]` 标记

**验收标准**：幻觉率 ≤ 10%

**文件**：`backend/rag/retriever.py`、`backend/rag/hybrid_search.py`

### 1.3 熔断器接入（D11）

**问题**：`CircuitBreaker` 已实现（19 个测试全过），但生产代码中零引用。

**修复方案**：在三个外部调用点接入：

| 调用点 | 文件 | 熔断参数 |
|--------|------|----------|
| LLM API 调用 | `model_router.py:chat()` | 5 次失败→OPEN, 30s 冷却 |
| AKShare 数据源 | `mcp/tools/stock_price.py` | 3 次失败→OPEN, 60s 冷却 |
| ChromaDB 查询 | `hybrid_search.py` | 5 次失败→OPEN, 30s 冷却 |

**验收标准**：连续失败 5 次后快速返回错误（<10ms），不再重试

**文件**：`backend/rag/model_router.py`、`backend/mcp/tools/`、`backend/rag/hybrid_search.py`

## 🟡 阶段二：体验与功能补全（预计 1.5h）

### 2.1 前端 SSE 流式消费（D10）

**问题**：后端 `agent/analyze/stream` 已实现完整 SSE 事件流（plan_start → task_start → task_complete → chart → report_start → done），前端只用同步接口，用户等 16 秒看白屏。

**修复方案**：
- 在 `PresetAnalysis.tsx` 的分析按钮中切换到流式调用
- 新增 `useAnalysisStream` hook：管理 SSE 连接、进度状态、自动重连
- UI 展示：进度条 + 当前子任务名称 + 已完成/总数计数

**SSE 事件映射**：
```
plan_start     → 显示"正在规划分析步骤..."
task_start     → 显示"正在执行：{任务名}"
task_complete  → 进度条步进
chart          → 显示"正在生成图表..."
report_start   → 显示"正在生成报告..."
done           → 跳转到报告页
```

**验收标准**：分析过程中能看到逐步进度，不再白屏等待

**文件**：`web/src/pages/PresetAnalysis.tsx`、`web/src/hooks/useAnalysisStream.ts`（新建）

### 2.2 CI 测试全覆盖

**问题**：22% 后端测试（6 个文件，~82 个测试）在 CI 中被跳过，原因是缺少 pymupdf/sentence_transformers/chromadb。

**修复方案**：

方案 A：CI 安装重依赖（需下载模型，可能超时）
方案 B：用 mock 解除重依赖（在测试文件中处理）

推荐方案 B——检查被跳过的 6 个测试文件，对其中可 mock 的部分（test_integration、test_corpus_monitor、test_text_utils）加入 CI，真正需要模型的（test_pdf_loader、test_hybrid_search、test_table_extraction）保持排除但做好标注。

**验收标准**：CI 测试数从 295 → 320+

**文件**：`.github/workflows/test.yml`、`requirements-ci.txt`

### 2.3 Docker 一键部署验证

**问题**：`docker-compose.yml` + `Dockerfile.backend` + `Dockerfile.frontend` 已存在但从未实际跑过。

**修复方案**：
- 修复 `Dockerfile.backend`：确认模型路径映射、环境变量、依赖安装顺序
- 验证 `docker-compose up` 能正常启动三服务（Redis + Backend + Frontend）
- 添加健康检查脚本 `scripts/docker-health.sh`

**验收标准**：`docker-compose up` → 浏览器访问 `localhost:80` 能看到完整应用

**文件**：`docker-compose.yml`、`Dockerfile.backend`、`Dockerfile.frontend`、`nginx.conf`

## 🟢 阶段三：锦上添花（预计 1h）

### 3.1 前端 E2E 测试

**问题**：0 个端到端测试，改前端可能悄悄坏后端。

**修复方案**：添加 3 个核心流程 Playwright E2E 测试：
1. 预设分析流程：选公司→选模板→点分析→看到报告
2. 文档上传流程：上传 PDF→选文档→提问→看到回答+来源
3. 跨页面导航：分析结果在预设分析和报告页之间保持

**验收标准**：3 个 E2E 全过

**文件**：`web/e2e/`（新建）

### 3.2 Agent 超时策略优化

**问题**：Agent 15 题中 3 题超时（>30s），平均 16.1s。

**修复方案**：
- Agent 单任务超时从 30s 降到 15s
- Planner 限制最多拆 5 个子任务（当前 10）
- 简单查询（单公司单指标）走 SQL 直查，不经过 Agent

**验收标准**：0 超时，平均 <10s

**文件**：`backend/agent/graph.py`、`backend/agent/planner.py`、`backend/config.py`

### 3.3 口语化查询支持（L1）

**问题**："赚了多少钱"→查不出来。

**修复方案**：
- 在 `query_processor.py` 的术语展开表中添加口语化映射：
  - "赚了多少钱" → "净利润"
  - "欠了多少钱" → "总负债"
  - "值多少钱" → "总市值/净资产"
  - "成本多少" → "营业成本"

**验收标准**：口语化 query 能匹配到正确指标名

**文件**：`backend/rag/query_processor.py`
